Narrative about the database
in my capstone project

The inclusion of the database was one of the primary steps I took while developing and designing my capstone project. The design changed throughout the development due necessities by the UI. For instance, I had to add another array for registered classes to the student’s record collection because I forgot about the register student functionality. But other than the slight changes over time I started designing and developing the database from week one to week six.

The reason to add this into my capstone project is the need to store student, class, admin, etc. information somewhere. The information just seemed best to be stored in a database. Since this is my first time developing a database from beginning to end, I think this portion of the capstone project shows my ability to adapt. I have implemented and even designed my own databases, but prior to this had not done both.

After I completed the development and design of the database, I have a good understanding of how important it is to properly organize and structure a database. I found in the early days I had a poorly managed database and ti was lacking in multiple ways, but then I forced myself to do a complete redesign of the database and that cleaned up a lot of my issues.

Design

I broke my database into 6 collections: Admins, Classes, Departments, Majors, Students, and Users. Admins, classes, and Students will be described more in-depth later.

The most important thing that I had to worry about when designing my database was the ability for my UI to interact with the database without much difficulties. I wanted query time to be limited to prevent UI freezing / stalling for information so I broke apart the information as much as I could.

A common theme that I implemented was the use of identifications for searching. I added in student ID, admin ID, and class ID to provide me with the ability to link the different collections. With giving each class an identification, I could easily link the current class a teacher is teaching without having to duplicate all the class information in the database.

Another common theme I took was to create sub documents in my collections. For instance, Student has a sub document of type AcademicInformation. The reason that I use sub documents is to allow for easier reading and extrapolation of data in the collection.

Student Collection

There were some requirements given in the previous class about the information stored in the student record, but I was given a lot of freedom in the execution. Some information I was required to have were student name, identification, and other personal information

The reason that I broke a student’s classes into three arrays is it allows me to move around classes when semesters are starting; as well as, maintaining the structure I still have when the switch occurs. When we move from semester A to semester B Enrolled classes become ClassesTaken and Registered classes become Enrolled. Each of the three arrays contains the same basic object: ClassName and grade (As you can see in the second image).

Admin Collection

Inside of the Admin collection it contains personal information and the classes the Admin is currently teaching. The current classes that the admin is teaching is an array that contains multiple instances of CurrentClass. The teaching class array gives me the flexibility to assign multiple classes to each teacher without having to worry about another object container.

The CurrentClass object within the admin collection is most of the interesting characteristics of the admin objects. The reason for this is the linking done through the ClassID and the StudentID. A CurrentClass will contain a list of students who are currently enrolled in the course. This allows the Admin to edit a student’s information without having to query the database each time. The reason that it contains ClassID was to, in future iterations, allow for another collection, called Registered, where it would link student’s registration and the classes they registered for. Sadly, I could not get to that.

Class Collection

The final important collection is the Class collection. Inside of the class collection we will contain 2 types: Online and Face-To-Face classes. In Online classes it contains class Url, begin date, and end date. In Face-To-Face class object it contains classroom and class building; as well as, begin and end date. Both use the course information object.

The reason that both online and Face-To-Face contain being and end dates is because that is information pertaining explicitly to a class, whereas, course information could be applied to multiple classes. For instance, if we had a class being taught over multiple semesters. This is also why ClassID is contained within course, for future iterations I could allow for semester based storage and link the classes with the course information by the ClassID (Should have been named CourseID).
